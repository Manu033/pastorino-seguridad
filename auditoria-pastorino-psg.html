Resumen Ejecutivo

La aplicacion presenta riesgos relevantes para confidencialidad, control de acceso y exposicion de infraestructura. El problema mas urgente es que el login opera por HTTP plano: credenciales y cookies pueden ser interceptadas en transito. Tambien se confirmo acceso sin autenticacion a pantallas internas sensibles, incluyendo permisos y estado de sincronizacion.

La prioridad debe ser bloquear el acceso anonimo a controladores internos, habilitar HTTPS valido, endurecer cookies y retirar cabeceras que revelan versiones del stack.

No se realizaron pruebas destructivas, fuerza bruta, explotacion activa ni cambios de datos.

Hallazgos Criticos

Critico

Login y sesion transmitidos por HTTP plano

El sitio responde por http:// y la conexion HTTPS en el puerto 443 resetea la conexion. Esto expone usuario, contrasena y cookies de sesion a captura en redes intermedias.

Evidencia
  • http://pastorinoseguridad.ddns.net/psg/backend devuelve 200 OK.
  • https://pastorinoseguridad.ddns.net/psg/backend falla con reset de conexion.
  • Cookies observadas: ASP.NET_SessionId, __RequestVerificationToken... y .ASPXAUTH sin atributo Secure.

Remediacion: instalar certificado TLS valido, forzar redireccion HTTP a HTTPS, activar HSTS y marcar cookies como Secure, HttpOnly y SameSite=Lax o Strict.

Critico

Rutas internas accesibles sin autenticacion

Se confirmaron paginas internas que responden 200 OK sin sesion autenticada. Esto permite ver pantallas de administracion y configuracion desde internet.

Evidencia
  • /psg/backend/MenusRoles/Index devuelve 200 OK sin cookie de autenticacion y muestra la pantalla PSG - Permisos.
  • /psg/backend/Sistema/ControlSincroNube devuelve 200 OK sin cookie de autenticacion y muestra PSG - Control SyncData.
  • La pantalla de permisos expone roles como Administrador, Cobranzas, Ventas, RRHH - Personal y endpoints JavaScript como /menusroles/GuardarRol y /menusroles/EliminarRol.

Remediacion: aplicar autorizacion por defecto a nivel global o controlador, por ejemplo [Authorize] en ASP.NET MVC, y dejar anonimo solo el login y recursos publicos estrictamente necesarios.

Critico

Credencial debil y expuesta

La credencial provista para la auditoria es simple y fue compartida en texto claro. Debe considerarse comprometida.

Evidencia
  • El login con el usuario provisto emitio cookie .ASPXAUTH y redirigio a /psg/backend/Home/Index.
  • La contrasena tiene baja complejidad y no debe mantenerse activa.

Remediacion: cambiar la contrasena inmediatamente, invalidar sesiones activas, revisar accesos recientes y aplicar politica de contrasenas robusta con MFA si el sistema lo permite.

Hallazgos Altos

Alto

Divulgacion de tecnologia y versiones

El servidor publica cabeceras que identifican plataforma y versiones, lo que facilita busqueda de vulnerabilidades especificas.

Cabeceras observadas
  • Server: Microsoft-IIS/8.5
  • X-AspNetMvc-Version: 5.2
  • X-AspNet-Version: 4.0.30319
  • X-Powered-By: ASP.NET

Remediacion: remover cabeceras de version desde IIS y configuracion ASP.NET. Mantener sistema operativo, IIS, .NET Framework y dependencias actualizadas.

Alto

Error 500 accesible sin sesion

La ruta /psg/backend/Account/Manage devuelve 500 Internal Server Error sin autenticacion. Esto sugiere validacion insuficiente de sesion o manejo incompleto de errores.

Evidencia
  • GET /psg/backend/Account/Manage sin sesion devuelve 500 Internal Server Error.
  • La respuesta renderiza el layout interno antes del mensaje de error.

Remediacion: validar autenticacion antes de ejecutar la accion, devolver 302 al login o 401/403, y registrar el error sin exponer layout interno.

Alto

Cabeceras de seguridad incompletas

La aplicacion no devuelve varias cabeceras de endurecimiento habituales. Esto aumenta la exposicion ante XSS, clickjacking parcial, MIME sniffing y filtrado de informacion.

Faltantes recomendados
  • Content-Security-Policy
  • X-Content-Type-Options: nosniff
  • Referrer-Policy
  • Permissions-Policy
  • Strict-Transport-Security, una vez activo HTTPS

Remediacion: definir politica CSP compatible con la aplicacion, agregar cabeceras desde IIS/web.config y revisar efectos en QA antes de publicar.

Hallazgos Medios

Medio

Librerias JavaScript antiguas y duplicadas

Se observaron multiples versiones de jQuery cargadas en paginas internas, incluyendo versiones antiguas con historial de vulnerabilidades conocidas.

Evidencia
  • jQuery v1.9.1 en /Content/flatty/assets/javascripts/jquery/jquery.min.js.
  • jQuery v3.3.1 en /Content/custom/js/jquery-3.3.1.js.

Remediacion: unificar version de jQuery, actualizar plugins dependientes y retirar librerias demo o no utilizadas.

Medio

Metodos HTTP no restringidos

El servidor responde 200 OK para OPTIONS y TRACE. Aunque no se observo eco clasico de TRACE en la prueba directa, conviene reducir la superficie expuesta.

Evidencia
  • OPTIONS respondio 200 OK.
  • TRACE respondio 200 OK.

Remediacion: bloquear metodos no usados desde IIS y permitir solo GET, POST y HEAD donde corresponda.

Plan de Remediacion Priorizado

  1. Activar HTTPS valido, redirigir HTTP a HTTPS y habilitar HSTS.
  2. Cambiar la contrasena auditada, invalidar sesiones y revisar logs de acceso.
  3. Aplicar autenticacion obligatoria por defecto en controladores internos.
  4. Corregir especificamente el acceso anonimo a MenusRoles y Sistema/ControlSincroNube.
  5. Marcar cookies como Secure, HttpOnly y SameSite.
  6. Ocultar cabeceras Server, X-AspNetMvc-Version, X-AspNet-Version y X-Powered-By.
  7. Agregar cabeceras de seguridad: CSP, nosniff, Referrer-Policy y Permissions-Policy.
  8. Actualizar librerias JavaScript y eliminar duplicados.
  9. Bloquear metodos HTTP no necesarios.
  10. Repetir prueba no destructiva luego de aplicar cambios.

Limitaciones

  • No se audito codigo fuente ni configuracion interna del servidor.
  • No se validaron endpoints POST sensibles para evitar cambios en produccion.
